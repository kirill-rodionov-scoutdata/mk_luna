consumerretryhttpx
▼

After status update, POST to payment.webhook_url with result payload.

  • Use httpx.AsyncClient for async HTTP calls
  • Retry schedule: attempt 1 → wait 2s → attempt 2 → wait 4s → attempt 3
  • Payload: {"payment_id", "status", "amount", "currency", "processed_at"}
  • If all 3 attempts fail: log error, do NOT raise (payment is already processed)
  • Timeout per request: 5 seconds
Helper: async def send_webhook(url, payload, max_retries=3)
Back-off: await asyncio.sleep(2 ** attempt)
Dead Letter Queue — NACK after 3 processing failures

If the consumer raises an unhandled exception, FastStream NACKs the message. After 3 NACKs RabbitMQ routes it to the DLQ (already configured in broker.py).

  • Wrap entire handler in try/except; re-raise only on unexpected infra errors

Phase 4 — Outbox Relay (Guaranteed Delivery)

Pending
Background outbox relay loop
outboxbackground task
▼

The Outbox pattern ensures payments are atomically saved to DB and queued for the broker — even if RabbitMQ is temporarily down.

  • Create src/app/infra/messaging/outbox_relay.py
  • Loop: while True: poll DB for unpublished rows → publish → mark published → sleep(1)
  • Run as asyncio.create_task() inside FastAPI lifespan (api service)
  • Use UnitOfWork to read and mark rows atomically
  • Publish via RabbitMQEventPublisher.publish(row["event_type"], row["payload"])
New file: src/app/infra/messaging/outbox_relay.py
Wired in: src/app/main.py → lifespan asyncio.create_task
Verify atomic write: payment + outbox row in same transaction
outboxUoW
▼
  • Both uow.payments.add() and uow.outbox.add() happen before await uow.commit()
  • Already wired in PaymentService.create_payment() — verify no commit between them
  • If broker is down during relay, outbox row stays unpublished; next poll picks it up
